# Condensation Library

The condensation library is the metadata and expansion substrate for compact
LLM-facing forms. Its first layer is deliberately small: helpers store source,
expansion, generated symbols, contracts, laws, checker status, and dynamic
frontier information on ordinary symbol property lists.

## Metadata Keys

Condensation metadata uses string property-list keys:

| Key | Meaning |
|-----|---------|
| `"condense.kind"` | Kind of condensed artifact, such as `concept` |
| `"condense.source"` | Original condensed source form |
| `"condense.expansion"` | Expansion generated from the source |
| `"condense.generated"` | Symbols generated by the expansion |
| `"condense.contracts"` | Local contracts attached to the artifact |
| `"condense.laws"` | Laws or examples attached to the artifact |
| `"condense.check-status"` | Checker result, such as checked/native/dynamic |
| `"condense.dynamic-frontier"` | Any dynamic or operative boundary crossed |
| `"condense.examples"` | Executable examples attached to a concept |

## Helpers

```lisp
(condense-put sym key value)
(condense-get sym key)
(condense-record! sym kind source expansion generated)
(condense-trace sym)
(condense-check-type sym)
```

`condense-trace` returns an alist so later tools can inspect the artifact without
parsing printed text.

```lisp
(condense-record!
  'invoice
  'concept
  '(defconcept invoice ...)
  '(progn ...)
  '(make-invoice invoice-p validate-invoice))

(condense-trace 'invoice)
```

Higher-level forms such as `defconcept`, `derive`, and capability dictionaries
should build on this metadata layer instead of creating private conventions.

`condense-check-type` runs `check-type` over generated symbols and stores
per-symbol results in `"condense.check-status"`. Results classified as `dynamic`
or `type-error` are also stored in `"condense.dynamic-frontier"` so condensed
forms do not pretend unchecked generated code was verified.

## `defconcept` v0

`defconcept` defines a compact tagged-list concept and records its expansion:

```lisp
(defconcept invoice
  (:fields ((id int64)
            (amount int64)
            (status symbol)))
  (:invariant (>= amount 0)))
```

The v0 representation is a tagged list:

```lisp
(make-invoice 1 100 'draft)
; => (INVOICE 1 100 DRAFT)
```

Generated names are deterministic:

```lisp
make-invoice
invoice-p
invoice-id
invoice-amount
invoice-status
validate-invoice
```

The optional `:invariant` form is checked inside `validate-<concept>` with field
names locally bound to accessor results. This is intentionally dynamic Lisp in
v0; later versions can lower supported concepts into typed structs or checked
applicative functions.

## `derive` v0

`derive` reads concept metadata and generates deterministic support functions:

```lisp
(derive invoice printer equality)
```

For `invoice`, v0 supports:

```lisp
invoice->plist
invoice-equal
```

`invoice->plist` returns an alist-like list of field/value pairs. `invoice-equal`
requires both arguments to satisfy `invoice-p` and then compares the whole tagged
representation with `equal`.

## Laws And Examples

`deflaw` attaches an inspectable predicate to a concept:

```lisp
(deflaw invoice-nonnegative
  (:for invoice)
  (:assert (>= amount 0)))
```

The law function accepts `self`, checks that it is an `invoice`, locally binds
field names, and evaluates the assertion.

`example` attaches an executable zero-argument check. The `:given` value is bound
to `*it*` while the `:expect` expression runs:

```lisp
(example valid-draft-invoice
  (:for invoice)
  (:given (make-invoice 1 100 'draft))
  (:expect (validate-invoice *it*)))
```

`condense-check` runs examples attached to a concept and returns
`(PASS . RESULTS)`:

```lisp
(condense-check 'invoice)
; => (T (VALID-DRAFT-INVOICE . T))
```
