# 14. OS Integration

This chapter covers typed access to the host operating system: process
identity and environment variables, wall-clock and monotonic time,
randomness, and spawning/controlling child processes — two optional
embedded libraries, `OS` (`lib/41-os.lisp`, portable) and `OS-LINUX`
(`lib/42-os-linux.lisp`, typed Linux-specific facilities), built entirely
on `std::env`/`std::process`/`std::fs`/`std::time` with zero new crate
dependencies. Pull them in with `(require 'os)`/`(require 'os-linux)` on a
`with_prelude()`-style environment; `with_stdlib()` environments (including
the `lamedh` CLI) already have both loaded.

**There is no `(syscall number ...)` primitive, and Lisp code never sees a
bare file descriptor or a bare PID as authority.** A spawned process is an
opaque `OS:CHILD` handle — it compares by identity, closes/reaps
deterministically, and has a Drop backstop (Chapter 11's ownership model,
applied to processes) — never something reconstructible from a printed
integer. Its stdio pipes, when requested, are ordinary `PORTS` ports
(Chapter 11): every `read-byte!`/`write-string!`/`close!` operation already
works on them. Signals are sent only by typed name (`:term`, `:kill`,
`:hup`, ...), never a raw signal number.

## 14.1 Capabilities

Four ambient authorities, checked the same `with-capabilities`-attenuable
way as `READ-FS`/`NET-CONNECT` (Chapter 7):

| Capability | Gates |
|---|---|
| `OS-ENV` | Reading process identity/environment: `os:args`, `os:executable-path`, `os:cwd`, `os:env-get`, `os:env-list`, `os:pid`, `os:ppid`, `os:hostname` |
| `OS-ENV-WRITE` | Mutating it: `os:chdir!`, `os:env-set!`, `os:env-unset!` |
| `OS-PROCESS` | Spawning a child: `os:spawn` |
| `OS-SIGNAL` | Signaling a PID not held as an owned child handle: `os:signal!` |

```console
$ target/debug/lamedh -s "(os:pid)"
Error: OS-ENV capability is not enabled (grant it via --capability OS-ENV or the host API)
  in: OS:PID
```

Once `os:spawn` returns a child handle, using *that* handle needs no
further capability check — `os:process-wait!`, `os:process-try-wait!`,
`os:process-kill!`, `os:process-terminate!`, `os:process-id`, and
`os:process-alive-p` all work under `OS-PROCESS` alone (Chapter 11's "an
open handle is authority to keep using it" rule). Signaling a PID you did
*not* get back from `os:spawn` — an arbitrary integer PID — is the separate
`OS-SIGNAL` authority.

`os-linux:stat`/`os-linux:lstat`/`os-linux:readlink` reuse the existing
`READ-FS` capability (Chapter 7) rather than inventing a parallel one —
they are filesystem metadata reads.

Time (`os:now`, `os:monotonic-nanos`, `os:sleep`) and randomness
(`os:make-prng`/`os:prng-next`, `os:random-bytes`) need **no capability** —
pure or read-only-entropy operations, the same treatment the pre-existing
global `(random n)` already gets.

Every capability here is independent and fence-attenuated exactly like the
rest of the sandbox:

```console
$ target/debug/lamedh --capability OS-PROCESS -s "(with-capabilities '() (os:spawn \"/bin/true\"))"
Error: capability denied: OS-PROCESS (attenuated by an enclosing fence)
  in: OS:SPAWN
```

## 14.2 Process identity and environment

```console
$ target/debug/lamedh --capability OS-ENV -s "(os:pid)" -s "(os:cwd)" -s "(car (os:args))"
638274
"/home/user/project"
"target/debug/lamedh"
```

- `(os:args)` — the process's raw `argv` (including `argv[0]`) as a list of
  strings. This is distinct from `*ARGV*`, which the CLI binds to only the
  script arguments that follow a script path (see Chapter 1).
- `(os:executable-path)` — the absolute path to the running executable.
- `(os:cwd)` / `(os:chdir! path)` — read/change the working directory
  (`OS-ENV-WRITE` for the write side). This is process-wide state: changing
  it affects every subsequent relative-path filesystem operation.
- `(os:env-get name)` — a string, or `NIL` if unset.
- `(os:env-list)` — every variable as an alist of `(name . value)` strings.
- `(os:env-set! name value)` / `(os:env-unset! name)` — `OS-ENV-WRITE`.
- `(os:pid)` / `(os:ppid)` — this process's PID / its parent's PID.
  `os:ppid` reads Linux's `/proc/self/stat` (std has no portable
  `getppid()`); on a non-Linux target it signals a structured
  `:unsupported-platform` error rather than binding opaquely.
- `(os:hostname)` — reads `/proc/sys/kernel/hostname` for the same reason.

## 14.3 Time

```console
$ target/debug/lamedh -s "(os:now)"
(1783849541 . 722806017)
```

`(os:now)` returns `(seconds . nanoseconds)` since the Unix epoch;
`(os:now-unix)` folds that into a single float. `(os:monotonic-nanos)`
returns nanoseconds since an arbitrary, process-local, never-goes-backward
reference point — not comparable across processes or with `os:now` — and
`(os:elapsed-seconds start)` turns a difference between two
`os:monotonic-nanos` readings into a float number of seconds:

```lisp
(let ((start (os:monotonic-nanos)))
  (do-work)
  (os:elapsed-seconds start))
; => 0.042 (however long do-work took)
```

`(os:sleep ms)` / `(os:sleep-seconds secs)` block the calling thread.
`std::thread::sleep` always sleeps for at least the requested duration (it
retries internally on spurious wakeups), so no `EINTR`/short-sleep behavior
is observable from Lisp.

## 14.4 Randomness

Deterministic and secure randomness are distinct APIs — neither touches or
replaces the pre-existing global `(random n)` (a time-seeded convenience
function that stays exactly as it was).

`(os:make-prng seed)` / `(os:prng-next state)` are a pure, explicitly-seeded
SplitMix64 generator: `os:prng-next` takes a state and returns
`(new-state . value)` (`value` a non-negative integer in `[0, 2^63)`),
never mutating anything in place — pass the returned state to the next
call:

```lisp
(let* ((s0 (os:make-prng 42))
       (r1 (os:prng-next s0))
       (r2 (os:prng-next (car r1))))
  (list (cdr r1) (cdr r2)))
; => (6839728766377637706 1474913046063446145) -- deterministic for seed 42
```

`(os:random-double state)` is the same step as a float in `[0.0, 1.0)`.

`(os:random-bytes n)` reads `n` cryptographically secure bytes from the OS
entropy source (`/dev/urandom` on Linux) as a fresh `Array<Char>`. No
capability required — a read-only entropy source, not application data.

## 14.5 Spawning processes

```lisp
(os:spawn program &optional argv
          &key (inherit-env t) env cwd stdin stdout stderr)
```

`program` is a path — **never** run through a shell, so shell
metacharacters in `argv` are passed through literally, with no
interpolation/injection risk:

```console
$ target/debug/lamedh --capability OS-PROCESS -s \
  "(let* ((r (os:spawn \"/bin/echo\" (list \"a;b|c\") :stdout ':pipe)))
     (ports:read-line! (os:process-stdout r)))"
"a;b|c"
```

- `argv` — a list of strings, not including `program` itself.
- `:inherit-env` (default `t`) — when true, the child inherits this
  process's environment, with `:env`'s `(name . value)` pairs applied as
  overrides; when `nil`, the child's environment is *exactly* `:env`'s
  pairs (nothing inherited).
- `:cwd` — the child's working directory, or `nil` to inherit this
  process's.
- `:stdin`/`:stdout`/`:stderr` — each `nil`/`:inherit` (share this
  process's stream), `:null` (discard/no input), or `:pipe` (return a
  `PORTS` port).

`os:spawn` returns a structured alist, not a bare handle:

```lisp
((:handle . #<process ...>) (:stdin . nil-or-port)
 (:stdout . nil-or-port) (:stderr . nil-or-port))
```

Accessors: `os:process-handle`, `os:process-stdin`, `os:process-stdout`,
`os:process-stderr`.

```console
$ target/debug/lamedh --capability OS-PROCESS -s \
  "(let* ((r (os:spawn \"/bin/cat\" () :stdin ':pipe :stdout ':pipe))
          (h (os:process-handle r)))
     (ports:write-string! (os:process-stdin r) \"hi\")
     (ports:close! (os:process-stdin r))
     (let ((reply (ports:read-line! (os:process-stdout r))))
       (os:process-wait! h)
       reply))"
"hi"
```

## 14.6 Waiting, killing, and terminating

- `(os:process-wait! handle)` — block until the child exits, reap it, and
  return an exit-status alist: `((:exit-code . n-or-nil) (:signal . n-or-nil)
  (:success . t-or-nil))`. Idempotent: calling it again after the child is
  already reaped returns the same cached status.
- `(os:process-try-wait! handle)` — non-blocking: `nil` if still running,
  else the same exit-status alist.
- `os:exit-code`, `os:exit-signal`, `os:exit-success-p` — accessors on an
  exit-status alist.
- `(os:process-kill! handle)` — `SIGKILL` (hard, unignorable). Does not
  reap; call `os:process-wait!` afterward, exactly like POSIX
  `kill(2)` + `waitpid(2)`.
- `(os:process-terminate! handle)` — `SIGTERM` (graceful; the child may
  ignore or handle it).
- `(os:process-alive-p handle)` / `(os:process-id handle)` — liveness and
  PID (the PID is retained for diagnostics even after the process is
  reaped).

Using a handle after it has been reaped is a structured `:closed` error,
not a crash or a silent no-op:

```console
$ target/debug/lamedh --capability OS-PROCESS -s \
  "(let* ((r (os:spawn \"/bin/true\")) (h (os:process-handle r)))
     (os:process-wait! h)
     (os:process-terminate! h))"
Error: terminate: CLOSED: process handle is closed (already reaped)
  in: OS:PROCESS-TERMINATE!
```

**Drop backstop**: if a child handle is garbage-collected without an
explicit `os:process-wait!`/`os:process-try-wait!`, a best-effort
non-blocking reap runs when the last reference goes away — releasing an
*already-exited* child's resources so it does not linger. A child that is
still *running* at that point is deliberately left running, not killed —
this is a safety net against leaks, not an implicit kill.

## 14.7 Signals

```lisp
(os:signal! pid signal-name)
```

`signal-name` is a typed name — `:term`, `:kill`, `:hup`, `:int`, `:usr1`,
`:usr2`, `:quit`, `:cont`, `:stop`, `:chld`, `:pipe`, `:alrm` (the `sig`
prefix is optional) — never a raw number. Requires `OS-SIGNAL`. Prefer
`os:process-kill!`/`os:process-terminate!` when you hold an `OS:CHILD`
handle for the target: those need only `OS-PROCESS` (continue-authority on
a resource you already hold), not the broader `OS-SIGNAL` grant needed to
reach an arbitrary PID.

## 14.8 The host OS policy hook

Embedding hosts have one more lever Lisp code cannot reach: a policy
callback, `Environment::set_os_policy`, that scopes a granted capability to
specific executables/argv/working directories or target PIDs — checked in
addition to, not instead of, the capability, mirroring
`set_net_policy` (Chapter 13). See `docs/embedding.md`'s "OS policy"
section for the Rust API and a worked example.

## 14.9 Structured errors

Every OS-module failure is a `LispVal::Error` whose data is an alist with
at least `:operation` (a string) and `:category` — one of `:not-found`,
`:permission-denied`, `:already-exists`, `:invalid-argument`, `:closed`,
`:policy-denied`, `:unsupported-platform`, `:signal-failed`, or `:other` —
plus `:os-error` (the underlying OS error text) where applicable:

```lisp
(handler-case (os:spawn "/no/such/binary")
  (error (e) (cdr (assoc ':category (error-data e)))))
; => :NOT-FOUND
```

## 14.10 OS-LINUX: file metadata and links

`(require 'os-linux)` loads a second, separately named module — deliberate,
so `(require 'os)` never silently pulls in platform-specific vocabulary.
The underlying Rust implementation (`std::os::unix`) is POSIX-portable
across every Unix std supports, but this module is scoped and named per the
epic's "Linux-only module" ruling; on a non-Unix target its primitives
signal a structured `:unsupported-platform` error rather than failing to
bind.

- `(os-linux:stat path)` — metadata following a trailing symlink (like
  `stat(2)`), as a typed alist: `((:size . bytes) (:mode . permission-bits)
  (:uid . n) (:gid . n) (:nlink . n) (:ino . n) (:dev . n)
  (:mtime . epoch-seconds) (:atime . epoch-seconds) (:ctime . epoch-seconds)
  (:is-dir . t-or-nil) (:is-file . t-or-nil) (:is-symlink . t-or-nil))` —
  never a raw C `struct stat`.
- `(os-linux:lstat path)` — like `stat`, but does not follow a trailing
  symlink (like `lstat(2)`).
- `(os-linux:readlink path)` — the target a symlink points to, as a string.
- Accessors: `os-linux:stat-size`, `-mode`, `-uid`, `-gid`, `-nlink`,
  `-ino`, `-dev`, `-mtime`, `-atime`, `-ctime`, `-directory-p`, `-file-p`,
  `-symlink-p`.

```console
$ target/debug/lamedh --capability READ-FS -s \
  "(os-linux:stat-directory-p (os-linux:stat \"/tmp\"))"
T
```

Standalone pipes not tied to a spawned child, terminal/PTY settings,
`eventfd`/`timerfd`/`signalfd`/`epoll`, Unix-domain sockets, and
`openat`-style directory-relative operations are explicitly deferred (the
ticket's own "only after the basic resource/polling model is stable"
sequencing) — not implemented here.

## 14.11 Relation to `shell`/`sh`

`lib/07-shell.lisp`'s `(shell cmd)`/`(sh cmd)` (unchanged, `SHELL`
capability) run a command line through `/bin/sh -c` and return
`(exit-code stdout stderr)` once the whole command finishes — a
convenient, coarse escape hatch. `os:spawn` is the typed layer this chapter
covers: explicit argv with no shell interpolation, explicit
environment/cwd, independently configurable stdio, and an owned handle
with non-blocking `os:process-try-wait!`, `os:process-kill!`, and
`os:process-terminate!` — none of which `shell` exposes. Use `shell`/`sh`
when a shell command line is genuinely what you want; use `os:spawn` for
everything else.
